# v2.0.1 - Week Rollover Cache Readiness Patch

**Date:** 2026-05-31
**Type:** Patch
**Canon reuse:** Yes - `canonVersion: v2` unchanged, no canon re-download
**Cache invalidation:** Yes - `cacheNamespace: v2.0.1` forces client-side cache refresh

## What Changed

- Week rollover cache readiness fix so current-week slices no longer mask historical week data.
- Version discipline formalization: five-field manifest, `cacheNamespace` / `canonVersion` separation, and `preparedAt` / `releasedAt` distinction.
- Preload UI copy now uses exact `appVersion`.
- Badge popover shows patch date and v2 release line date.

## Why

v2.0.0 shipped with runtime-version and canon-version identity conflated. Codex pushed an initial week rollover fix as code, but without bumping any version field, browsers had no cache invalidation signal. The fix was present in production, but clients could keep restoring stale v2 state.

v2.0.1 separates the runtime version (`appVersion: v2.0.1`) from the canon artifact identity (`canonVersion: v2`), so cache invalidation happens without re-downloading the v2 canon bundle.

## How To Verify

1. Clear browser caches once: `localStorage`, IndexedDB Limni stores, and browser Cache Storage.
2. Reload the app.
3. Confirm the badge popover reports `appVersion: v2.0.1`.
4. Verify May 25 historical week renders correctly.
5. Verify canon bundle was not re-downloaded when an existing `canonVersion: v2` bundle is already present.

## Open Issue Status

This patch addresses the rollover bug. Status moves to monitoring through the next week rollover and after the next major release. If the rollover symptom recurs after cache invalidation discipline is in place, escalate.
