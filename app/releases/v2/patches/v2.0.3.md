## Status

v2.0.3 is no longer tied to the clean14/data-verification lane. The accepted live
patch is the institutional seed runtime:

- closed active baseline is 15 receipt-backed weeks, from the immutable seed
  start through the week displayed as Jun 01 2026.
- the week displayed as Jun 08 2026 is visible as the current live overlay, but
  it is not counted as a closed certified week.
- Data, Performance, and Status read the same active-baseline contract.
- source-freeze and active-baseline certification have scheduler/materialization
  receipts and cron routes.
- release canon files under `releases/v2/canon/` remain immutable for this
  patch; runtime strategy shards and receipts carry the institutional-seed work.

This patch is a technical release baseline, not a claim that every v2.0.4
product issue is fixed.

## What Changed

- Added the app-truth control layer for active baseline, route readiness,
  scheduler/materialization run ledgers, selected-ledger receipts, and legacy
  path registration.
- Added source-freeze lifecycle code and scripts so weekly source readiness can
  be built and audited from the same contract used by the app.
- Added cron routes for source-freeze production and active-baseline
  certification, and registered them in the app-truth scheduler contract.
- Moved the Performance default history from the old seed/clean14 window to the
  rolling `active-baseline` window.
- Kept the legacy seed window available only as explicit compatibility/history
  mode.
- Made current live-week visibility explicit: Performance and Data can expose
  Jun 08 2026 as live/current while closed readiness remains 15 weeks.
- Fixed the current-week ADR Grid loss-count aggregation so flat/no-fill rows
  are not counted as losses.
- Cleaned app-facing runtime names away from clean14 and toward the
  institutional-seed namespace.
