# v2.0.2 - Kernel Phase 1/3 Canon Inventory, Shard Store, and Active Runtime Sync

**Date:** 2026-06-01
**Type:** Patch
**Canon reuse:** Yes - `canonVersion: v2` unchanged, no frozen canon artifact mutation
**Cache invalidation:** Yes - `cacheNamespace: v2.0.2` marks the new kernel contract surface

## What Changed

- Added `docs/architecture/KERNEL_DATA_ARCHITECTURE_SPEC.md`, defining the three-layer data model:
  release canon, closed-week delta, and live week.
- Added a read-only canon inventory endpoint:
  `/api/canon/v2/inventory`.
- Added a read-only one-week shard endpoint:
  `/api/canon/v2/week?strategyVariant=...&weekOpenUtc=...`.
- Added server-only deterministic shard derivation from existing monolithic v2 canon artifacts.
- Added server-derived closed-week delta support:
  - inventory `deltaWeeks` are derived from DB-backed closed-history rows after the immutable release baseline,
  - `/api/canon/v2/week` falls back to a `closed-week-delta` shard for closed weeks not present in release canon,
  - current/open week remains excluded from delta shards.
- Reserved aggregate shard metadata for future path-dependent all-time metrics.
- Added IndexedDB v2 shard stores beside the legacy monolithic `bundles` and `meta` stores:
  `weekShards`, `inventories`, and `kernelMeta`.
- Added pure inventory gap helpers for comparing local shard records against server inventory.
- Added focused tests proving deterministic week-shard derivation, aggregate-slot inventory, missing/corrupt gap detection, and local inventory summaries.
- Added a client-side active-variant canon kernel store that:
  - loads the release manifest,
  - syncs cache namespace without deleting unchanged `v2` canon shards,
  - fetches the active-variant canon inventory,
  - diffs local IndexedDB shards against expected week hashes,
  - fetches only missing or corrupt active-variant week shards,
  - records kernel metadata and inventory state in IndexedDB.
- Added closed-history bundle composition from verified week shards and wired Basket closed-history snapshots to prefer the kernel-composed bundle with legacy monolithic canon as fallback.
- Updated the app preload gate so active Performance routes open on active-kernel readiness. Legacy monolithic canon remains a failure fallback for kernel routes and the normal gate for non-kernel routes, but it no longer runs as a delayed background download after kernel-ready Performance boot.
- Changed kernel-route strategy session boot to require the active strategy payload explicitly and skip the legacy all-strategy background preload queue. Non-strategy session domains may still refresh after the app opens; inactive strategy variants move to the future kernel background-variant queue.
- Added an explicit active strategy kernel payload endpoint:
  `/api/performance/strategy-kernel-payload`.
- Wired Performance to prefer the kernel payload endpoint while retaining the legacy `strategy-page-data` route for fallback and non-Performance consumers.
- Added a version badge diagnostic line:
  `Kernel: <status> (<ready>/<total> weeks, <rows> rows)`.
- Added Status page kernel diagnostics for version/cache/canon identity, active shard inventory, and the closed/live week boundary.
- Wired kernel diagnostics to the active Performance selection, and allowed `/status` to inspect a selected strategy variant through query params instead of a hidden hardcoded target.
- Stopped active kernel Performance boot from consulting the old broad strategy artifact status, repair, and global preload stamp path. Legacy repair remains available for non-kernel fallback paths only.
- Marked Matrix CFD/Crypto as intentionally provisional on the Status page for v2.0.2. Matrix is not part of the v2.0.2 readiness gate.
- Polished the Performance week strip so overflow arrows align on the same row, and constrained the version badge popover with an opaque shell plus internal scrollable release briefing list.
- Changed the version badge manifest request to `no-store` so local/released patch identity does not get stuck on a stale cached manifest.
- Unified Basket drilldown row context across all-time, closed-week, and current-week views so expanded week/tier/grid/trade levels show count and W/L context instead of changing behavior by period.
- Preserved current-week grid/fill detail in the active full payload so live ADR Grid Basket rows can expand into fills without preloading inactive variants. Single-fill current-week grids now render as `1 fill` with a `Fill 1` row instead of falling back to a flat `1 trade` detail.
- Normalized Basket fill display order so per-grid fills render as `Fill 1`, `Fill 2`, ... by entry order even when the underlying closed-week delta source sequence is global or sparse.

## Why

v2.0.1 contains the immediate rollover/cache-readiness patch. v2.0.2 starts the durable kernel track and adds non-blocking active-variant runtime sync without replacing the legacy render path yet.

The core problem is that v2 canon is currently stored as monolithic per-variant bundles. When a new week closes, the app has no granular inventory contract for deciding what it already has and what it is missing. Phase 1 introduces that contract while preserving immutability: existing files under `releases/v2/canon/` are read only and sliced on demand, while post-release closed weeks are served as `closed-week-delta` shards derived from the live DB.

## Scope Boundary

This patch partially replaces the current app preload gate for active Performance routes. The IndexedDB schema has been expanded safely beside the legacy monolithic stores, and the active strategy variant now syncs and composes closed-history through the kernel. Inventory requests are active-variant scoped so boot does not compute every variant's delta list. Basket closed-history snapshots prefer the kernel-composed bundle when available, with the legacy monolithic canon snapshot as fallback. Performance now prefers the explicit strategy kernel payload endpoint, which assembles from server-side strategy week shards and excludes the live/open week. Current-week payloads remain live-only. Kernel routes no longer start the old all-strategy preload queue or delayed monolithic canon download as part of boot.

`market-intelligence`, `news`, and `accounts` remain outside the kernel gate. The kernel track applies to strategy/canon/live-week data.

Matrix is intentionally degraded/provisional in v2.0.2. It should not be used as a readiness signal for the Performance data kernel or the upcoming indicator verification milestone.

   - `schemaVersion: canon-week-shard-v1`

14. Expand current-week ADR Grid pairs and confirm both single-fill and multi-fill live grids expose fill rows and fill details.

## Open Issue Status

This patch does not close the current v2 open issues by itself. It creates the Phase 1/3 contract, local storage groundwork, and active-variant runtime sync that the full kernel will use to close them:

- Week rollover issue: inventory now includes closed-week delta shards and the client can fetch only the missing delta week. Full closure still requires browser verification at the next rollover and full strategy payload kernel composition.
- Current-week Basket issue: current/open week remains live-only, but the live renderer now preserves active-week grid details and matches the closed-history drilldown language.
- Preloader loop issue: active Performance boot now gates only active kernel readiness plus active strategy kernel payload readiness; inactive strategy variants are no longer queued by the legacy background preloader on kernel routes, and active kernel boot no longer consults the legacy repair/stamp path. Full closure still requires replacing the remaining legacy fallback paths outside active Performance.
